# KerryWeb3Guard Telegram 渠道增长计划

> Project Creator: Telegram `@kerryzheng`

本计划用于指导 KerryWeb3Guard 从 Web MVP 走向 Telegram 场景分发，
在真实 Web3 社群中验证产品价值并建立可持续增长路径。

---

## 1. 计划名称与目标

**计划名称**：Telegram 渠道增长计划（Telegram Distribution Plan）

**核心目标**：

- 在 Telegram 场景拿到第一批真实用户
- 验证 3 个核心能力的使用频率与留存
- 建立低成本、可复制的社群分发机制

**阶段目标**：

- 上线 Bot v1（支持合约 / 链接 / 聊天三类分析）
- 接入 20 个 Web3 相关群或频道进行内测触达
- 累计 300+ 独立用户，7 日留存达到 20%+

---

## 2. 为什么优先 Telegram

- Web3 用户天然在 Telegram 活跃，分发路径短
- 使用场景高频：别人发链接、发合约、发聊天截图时可立即调用
- 与浏览器插件相比，上线更快、试错成本更低

**结论**：短期 Telegram 优先，插件作为中期计划。

---

## 3. 用户路径（核心漏斗）

1. 用户在群里看到 Bot 介绍或转发
2. 用户点击 `/start`
3. 用户选择检测类型（Contract / Link / Chat）
4. 用户输入内容并拿到风险分析结果
5. 用户转发结果给他人（形成传播）
6. 用户二次使用（留存）

---

## 4. Bot 功能范围（v1）

### 4.1 基础指令

- `/start`：欢迎与功能入口
- `/help`：使用说明
- `/lang`：中英文切换
- `/contract`：链上合约风险扫描
- `/link`：链接安全检测
- `/chat`：诈骗话术分析

### 4.2 输出结构（统一）

- 风险分数（0-100）
- 数据视图（可解释指标）
- AI 结论（Markdown 格式）
- 行动建议（Do / Don't）

### 4.3 v1 不做

- 用户系统与复杂权限
- 支付与订阅
- 高频并发优化（先验证需求）

## 5. 技术接入方案（基于现有后端）

Bot 不重复造轮子，直接复用现有 API：

- `POST /api/v1/scan/contract`
- `POST /api/v1/scan/link`
- `POST /api/v1/scan/chat`

建议架构：

- Telegram Bot（Python）-> KerryWeb3Guard Backend API -> LLM/Data Providers

建议新增环境变量（给 Bot 项目）：

- `TELEGRAM_BOT_TOKEN`
- `API_BASE_URL`
- `DEFAULT_RESPONSE_LANGUAGE`

### 双 Bot（中英文）复用方案

如果需要中英文分别的 Bot，但不希望维护两套代码，推荐：

- **同一代码仓**：`bot/telegram-bot`
- **两个 Bot token**：中文 Bot / 英文 Bot
- **两个 env 文件**：
  - `.env.zh`（`DEFAULT_RESPONSE_LANGUAGE=zh-CN`）
  - `.env.en`（`DEFAULT_RESPONSE_LANGUAGE=en`）

运行方式：同一份代码启动两个进程（分别加载不同 env）。

优点：

- 代码完全复用
- 文案和默认语言可分离
- 后续可独立运营中文与英文渠道

### Start 内容来源与维护位置

- 防骗指南与正规公司做法维护文件：  
  `backend/risk-service/app/core/anti_scam_guide.py`
- Bot `/start` 推荐调用接口：  
  `GET /api/v1/meta/start-guide?language=zh-CN`
- 兼容旧版纯列表接口：  
  `GET /api/v1/meta/anti-scam-guide?language=zh-CN`

维护方式：

1. 在 `SCAM_KNOWLEDGE_BASE` 增加/修改骗术条目  
2. 在 `LEGIT_COMPANY_PRACTICES` 增加/修改正规做法  
3. 每条内容都补 `source` 字段，便于说明依据

## 6. 渠道执行计划（阶段制）

## 阶段 A：准备与上线

- 完成 Bot 基础指令与三类检测接入
- 统一消息模板（短版/长版）
- 完成错误兜底与超时提示
- 小范围私测（5-10 人）

**验收**：

- 成功检测率 > 90%
- 单次响应时间可接受（P95 < 12s）

## 阶段 B：种子用户扩散

- 对接首批熟悉的 Web3 社群
- 发布固定演示案例（合约/链接/聊天）
- 采集用户反馈（误报、漏报、文案理解度）

**验收**：

- 达到第一批稳定独立用户
- 复用率（2 次以上）> 30%

## 阶段 C：增长优化

- 优化结果格式（更短、更易转发）
- 增加“快速按钮”与预设模板
- 引导用户分享检测结果到群内

**验收**：

- 用户规模与日均调用次数持续增长
- 三大模块调用占比更均衡

## 阶段 D：复盘与下一步

- 输出渠道复盘（哪些群效果最好）
- 制定插件版本需求文档（基于真实使用数据）
- 确定商业化试点方向（API 或 Pro 功能）

**验收**：

- 形成可复制的增长动作清单
- 形成明确的下一阶段产品路线

## 7. KPI 指标体系

**增长指标**：

- 新增用户数（Daily / Weekly）
- 来源群数量与转化率

**产品指标**：

- 检测调用总量
- 三类功能调用占比
- 成功率与超时率

**留存指标**：

- D1 / D7 留存
- 人均调用次数

**质量指标**：

- 用户反馈中的误报率（主观）
- “有帮助”反馈占比

## 8. 内容与传播策略

## 8.1 内容模板

- 模板 A：今日钓鱼链接案例（截图 + 检测结果）
- 模板 B：高风险合约案例（数据图 + 结论）
- 模板 C：诈骗话术拆解（对话片段 + 风险点）

## 8.2 传播机制

- 每条结果消息可一键转发
- 群管理员可设置“每日一条安全提示”
- 定期发布“本周诈骗案例 Top 5”

---

## 9. 风险与合规边界

- 明确声明：仅提供风险提示，不构成投资建议
- 不收集私钥、助记词、交易密码
- 用户输入内容默认最小化保存（或不落库）
- 黑名单内容需人工复核，保留申诉机制

---

## 10. 与浏览器插件的关系

Telegram 是当前增长入口，插件是下一阶段的场景深化：

- Telegram：低成本获客与模型迭代
- 插件：高价值拦截（连接钱包/签名前）

当 Telegram 达到稳定留存后，再启动插件开发最合理。
